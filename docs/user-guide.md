## 1. 什么是 Feature Flag

Feature Flag（功能开关）是一种在不重新发版的情况下，**动态控制功能是否对用户可见**的技术。

**典型用途：**

| 场景 | 说明 |
|------|------|
| 灰度发布 | 新功能先让 10% 用户试用，没问题再全量开放 |
| 紧急回滚 | 线上出 bug，一键关闭功能，不需要回滚代码 |
| 白名单测试 | 只对内部测试人员开放某个功能 |
| A/B 实验 | 给不同用户展示不同文案、不同 UI |
| 平台差异 | iOS 用户开放新功能，Android 暂不开放 |
| 版本控制 | 仅 2.0 以上版本开放新功能 |

**没有 Feature Flag 时：** 功能上线 = 代码发版，回滚 = 重新部署。
**有了 Feature Flag：** 功能上线 = 在后台翻一个开关，回滚 = 关掉开关。

## 2. FlagForge 核心概念

FlagForge 有 5 个核心概念，它们的层级关系是：

```
应用 (App)
  └── 环境 (Environment)      ← dev / staging / prod
  └── Feature (功能开关)
        └── 定向规则 (Targeting Rule)    ← 按优先级排序
              └── 条件 (Conditions)      ← 谁能命中这条规则
        └── 用户覆盖 (Override)          ← 最高优先级
```

### 2.1 应用 (App)

一个"应用"代表你的一个产品或服务。例如：

- `my_game` — 我的游戏
- `web_store` — 网上商城
- `internal_tools` — 内部工具

每个应用有独立的 `app_key`（唯一标识），客户端用这个 key 来拉取配置。

### 2.2 环境 (Environment)

同一个应用通常有多个运行环境：

| 环境 | 说明 | 典型用法 |
|------|------|---------|
| `dev` | 开发环境 | 开发人员本地调试 |
| `staging` | 预发布环境 | 上线前验证 |
| `prod` | 生产环境 | 实际用户使用 |
| `canary` | 金丝雀环境 | 小流量验证（可选） |

**为什么要分环境？** 你可能希望一个功能在 dev 环境开启，但在 prod 环境关闭。规则是按环境隔离的。

### 2.3 Feature (功能开关)

一个 Feature 对应你产品里的一个"可开关的功能"。每个 Feature 有：

- **key_name** — 唯一标识，客户端用它查询开关状态（如 `dark_mode`、`new_checkout`）
- **value_type** — 值类型：
  - `boolean`：仅 开/关
  - `string`：返回一段文字（如欢迎语、按钮文案）
  - `json`：返回结构化数据（如配置对象）

### 2.4 定向规则 (Targeting Rule)

规则决定了 **"谁在什么条件下，这个功能开或关、返回什么值"**。

一个 Feature 在一个环境下可以有多条规则。规则按 **priority（优先级）从小到大** 逐条匹配，**第一条命中的规则生效**，后续规则不再评估。

每条规则包含：
- **priority** — 数字越小越先匹配
- **conditions** — 匹配条件（谁能命中这条规则）
- **enabled** — 命中后 Feature 是开还是关
- **value** — 命中后返回的值（string/json 类型使用）

### 2.5 条件类型

FlagForge 支持 6 种条件：

| 条件类型 | 用途 | 示例 |
|---------|------|------|
| `user_list` | 用户白名单 | 仅 alice、bob 可见 |
| `percentage` | 灰度百分比 | 30% 的用户可见 |
| `version` | 版本约束 | 仅 ≥2.0.0 的客户端可见 |
| `platform` | 平台限制 | 仅 iOS 用户可见 |
| `attr` | 自定义属性 | 仅 region=cn 的用户可见 |
| AND/OR 组合 | 逻辑组合 | 白名单用户 AND iOS 平台 |

### 2.6 用户覆盖 (Override)

用户（或客户端）可以自己设置某个 Feature 的状态，**优先级高于所有规则**。例如用户在设置页里手动关闭了"暗色模式"，即使规则说应该打开，也以用户选择为准。

## 3. 启动服务

### 3.1 前置条件

| 软件 | 版本要求 | 安装方式（Windows） |
|------|---------|-------------------|
| Go | ≥ 1.24 | `winget install GoLang.Go` |
| Node.js | ≥ 20 | `winget install OpenJS.NodeJS.LTS` |
| Python 3 | ≥ 3.8 | `winget install Python.Python.3.12`（测试脚本用） |

### 3.2 启动后端

```bash
cd backend
go run .
```

后端运行于 http://localhost:8080，SQLite 数据库 `flagforge.db` 自动创建。

### 3.3 启动前端

```bash
cd web
npm install    # 首次需要
npm run dev
```

浏览器访问 http://localhost:5173 打开管理后台。

### 3.4 一键测试（可选）

```bash
pip install requests
python scripts/test_api.py
```

自动启动后端 → 运行 120 个测试用例 → 结束后关闭后端。

## 5. 第一步：创建应用

1. 点击左侧菜单 **应用管理**
2. 点击右上角 **+ 新建应用**
3. 填写信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 应用标识 | 唯一 ID，字母开头，仅含字母/数字/_/- | `my_game` |
| 应用名称 | 显示名称 | `我的游戏` |
| 描述 | 可选 | `一款休闲手游` |

4. 点击 **保存**

> ⚠️ **应用标识创建后不可修改**，请认真填写。客户端 SDK 通过 `app_key` 来拉取配置。

### 应用的统计卡片

创建后，应用管理页面会显示每个应用的环境数和 Feature 数，方便你快速了解整体情况。

---

## 6. 第二步：创建环境

1. 点击左侧菜单 **环境管理**
2. 在顶部 **下拉框** 中选择刚创建的应用
3. 点击 **+ 新建环境**
4. 填写信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 环境标识 | 唯一 ID | `dev` |
| 环境名称 | 显示名称 | `开发环境` |
| 排序 | 列表排序顺序（越小越靠前） | `0` |
| 生产环境 | 是否标记为生产（删除时会额外确认） | `否` |

**建议创建 3 个标准环境：**

| 标识 | 名称 | 排序 | 生产环境 |
|------|------|------|---------|
| `dev` | 开发环境 | 0 | 否 |
| `staging` | 预发布环境 | 10 | 否 |
| `prod` | 生产环境 | 20 | 是 |

> ⚠️ **环境标识创建后不可修改**。客户端通过 `env_key` 来指定环境。

---

## 7. 第三步：创建 Feature

1. 点击左侧菜单 **特性管理**
2. 在顶部选择 **应用** 和 **环境**
3. 点击 **+ 新建 Feature**

| 字段 | 说明 | 示例 |
|------|------|------|
| Feature Key | 唯一标识，客户端查询用 | `dark_mode` |
| 值类型 | boolean / string / json | `boolean` |
| 描述 | 可选 | `暗色模式开关` |

> ⚠️ **Feature Key 和值类型创建后不可修改**。

### 值类型选择指南

| 类型 | 适用场景 | 客户端使用方式 |
|------|---------|--------------|
| `boolean` | 纯开关（开/关） | `if (enabled) { 显示功能 }` |
| `string` | 文案/配置值 | `text = value`（如 `"欢迎回来"` / `"Welcome"`) |
| `json` | 结构化配置 | `config = JSON.parse(value)`（如 `{"color":"red","size":16}`） |

创建 Feature 后，它还没有任何规则。**没有规则 = 默认关闭**。接下来需要配置规则。

## 8. 第四步：配置规则

规则是 FlagForge 最核心的功能。有两种方式创建规则：

### 方式 A：从特性管理页进入（推荐）

1. 在特性管理页面，选择一个环境
2. 找到目标 Feature，点击该行的 **规则** 按钮
3. 在右侧弹出的抽屉中，点击 **新建规则**

### 方式 B：从规则管理页创建

1. 点击左侧菜单 **规则管理**
2. 选择应用和环境
3. 点击 **+ 新建规则**

### 规则配置项

| 字段 | 说明 |
|------|------|
| Feature | 选择要配规则的 Feature |
| 环境 | 选择生效的环境 |
| 规则名称 | 可选，便于辨认（如 "VIP 用户"、"基线规则"） |
| 优先级 | **数字越小越先匹配**（重要！） |
| 启用规则 | 开关，禁用后此规则不参与匹配 |
| 命中时开启 Feature | 命中此规则后，Feature 是开还是关 |
| 命中时返回值 | string/json 类型的 Feature 使用 |
| 条件 | 决定谁能命中这条规则 |

### 条件编辑器

条件编辑器支持添加多个条件，多个条件之间是 **AND（且）** 的关系。

#### 用户白名单 (user_list)

指定一组用户 ID，只有这些用户才能命中：

- 选择类型：`用户白名单 (user_list)`
- 输入用户 ID 后按回车（可以输入多个）

#### 灰度百分比 (percentage)

让一部分用户命中（基于用户 ID 哈希，相同用户每次结果一致）：

- 选择类型：`灰度百分比 (percentage)`
- 输入百分比（0-100）

#### 版本约束 (version)

根据客户端版本号匹配：

- 选择类型：`版本约束 (version)`
- 输入如 `>=2.0.0`（支持 `>` `>=` `<` `<=` `=`）

#### 平台限制 (platform)

根据客户端平台匹配：

- 选择类型：`平台 (platform)`
- 选择：iOS / Android / Windows / macOS / Linux / Web

#### 扩展属性 (attr)

根据自定义属性匹配（客户端传 `attr_region=cn`，这里匹配 `region = cn`）：

- 选择类型：`扩展属性 (attr)`
- 输入属性名（如 `region`）和属性值（如 `cn`）

### 理解优先级

规则按优先级从小到大逐条匹配，**第一条命中的规则生效**。

**举例：** Feature `dark_mode` 在 prod 环境下有 3 条规则：

| 优先级 | 规则名称 | 条件 | 命中时 | 说明 |
|--------|---------|------|--------|------|
| 0 | VIP 白名单 | user_list: alice, bob | 开启 | alice 和 bob 必定看到 |
| 10 | 灰度 30% | percentage: 30 | 开启 | 其余 30% 用户能看到 |
| 100 | 基线：默认关闭 | 空条件 (match all) | 关闭 | 兜底，所有人都命中 |

**求值过程（用户 = alice）：**
1. 检查规则 0 → alice 在白名单中 → **命中** → Feature = 开启 ✅

**求值过程（用户 = charlie）：**
1. 检查规则 0 → charlie 不在白名单 → 跳过
2. 检查规则 10 → hash(charlie) % 100 = 42，42 < 30? 不是 → 跳过
3. 检查规则 100 → 空条件 (match all) → **命中** → Feature = 关闭 ❌

**求值过程（用户 = dave）：**
1. 检查规则 0 → 跳过
2. 检查规则 10 → hash(dave) % 100 = 17，17 < 30? 是 → **命中** → Feature = 开启 ✅

> 💡 **最佳实践**：始终在最低优先级放一条空条件的"基线规则"作为兜底。否则无规则命中时 Feature 默认关闭且不返回值。

## 9. 第五步：验证效果

配置完规则后，用 **特性查询** 页面模拟客户端来验证。

1. 点击左侧菜单 **特性查询**
2. 选择应用（按 app_key）
3. 选择环境（按 env_key）
4. 输入 User ID（模拟谁在请求）
5. （可选）展开高级过滤，填写 version / platform / 自定义属性
6. 点击 **查询**

页面会显示所有 Feature 的求值结果：

- Feature Key
- 状态（开启/关闭）
- 返回值

**验证建议：** 多换几个 User ID 测试，确认白名单、灰度、兜底规则都符合预期。

## 10. 实战场景

### 场景 1：新功能灰度发布

**背景：** 你开发了一个新的结账流程 `new_checkout`，希望先让 5% 的用户试用，确认没问题后逐步放量到 100%。

**操作步骤：**

1. **创建 Feature**：
   - Key: `new_checkout`
   - 类型: `boolean`
   - 描述: `新结账流程`

2. **创建规则 1 — 内部测试白名单**（先让自己人测）：
   - 优先级: `0`
   - 条件: 用户白名单 → 输入测试人员的 User ID
   - 命中时: 开启

3. **创建规则 2 — 灰度 5%**：
   - 优先级: `10`
   - 条件: 灰度百分比 → `5`
   - 命中时: 开启

4. **创建规则 3 — 基线（兜底）**：
   - 优先级: `100`
   - 条件: 空（match all）
   - 命中时: 关闭

5. **验证**：用特性查询页面，分别用白名单用户和普通用户测试

6. **逐步放量**：确认稳定后，编辑规则 2，将百分比从 5 改为 20 → 50 → 100

7. **全量后清理**：把基线规则改为"开启"，删除灰度规则

---

### 场景 2：紧急关闭功能（Kill Switch）

**背景：** 线上功能 `payment_v2` 出现 bug，需要立即对所有用户关闭。

**操作步骤：**

1. 进入 **规则管理**，找到 `payment_v2` 在 `prod` 环境下的规则
2. **方法 A**：把最低优先级（数字最小）的规则改为：条件 = 空，命中时 = 关闭
3. **方法 B**：新建一条优先级 = `0` 的规则，条件 = 空，命中时 = 关闭（优先级最高，所有人命中）
4. **方法 C**：把所有"开启"的规则禁用（关闭"启用规则"开关）

> 💡 **推荐方法 B**，因为不修改已有规则，修复后只需删除这条临时规则即可恢复。

---

### 场景 3：iOS 先行，Android 稍后

**背景：** 新功能 `ar_camera` iOS 版本已就绪，Android 还在开发中。

**操作步骤：**

1. **创建 Feature**：Key = `ar_camera`，类型 = `boolean`

2. **创建规则 1 — iOS 用户开启**：
   - 优先级: `10`
   - 条件: 平台 → `ios`
   - 命中时: 开启

3. **创建规则 2 — 基线关闭**：
   - 优先级: `100`
   - 条件: 空
   - 命中时: 关闭

当 Android 也准备好了，新建一条规则：
- 优先级: `10`
- 条件: 平台 → `android`
- 命中时: 开启

---

### 场景 4：不同版本不同行为

**背景：** `welcome_text` 功能，2.0 以上版本显示新欢迎语，旧版本显示旧欢迎语。

**操作步骤：**

1. **创建 Feature**：Key = `welcome_text`，类型 = `string`

2. **创建规则 1 — 新版本**：
   - 优先级: `0`
   - 条件: 版本约束 → `>=2.0.0`
   - 命中时: 开启，返回值 = `欢迎使用全新版本！`

3. **创建规则 2 — 旧版本**：
   - 优先级: `100`
   - 条件: 空（match all）
   - 命中时: 开启，返回值 = `欢迎回来`

---

### 场景 5：基于地区的定向投放

**背景：** 促销活动 `spring_sale` 仅对中国区用户和美国区 VIP 用户开放。

**前提：** 客户端请求时需携带 `attr_region` 参数。

**操作步骤：**

1. **创建 Feature**：Key = `spring_sale`，类型 = `boolean`

2. **创建规则 1 — 中国区全量**：
   - 优先级: `10`
   - 条件: 扩展属性 → key = `region`，value = `cn`
   - 命中时: 开启

3. **创建规则 2 — 美国区 VIP**：
   - 优先级: `20`
   - 条件 1: 扩展属性 → key = `region`，value = `us`
   - 条件 2: 用户白名单 → VIP 用户 ID 列表
   - （两个条件之间是 AND 关系：美国区 **且** 在 VIP 名单中）
   - 命中时: 开启

4. **创建规则 3 — 基线关闭**：
   - 优先级: `100`
   - 条件: 空
   - 命中时: 关闭

---

### 场景 6：A/B 测试 — 不同用户看到不同文案

**背景：** 测试两种按钮文案的转化率。50% 用户看到 A 方案，50% 看到 B 方案。

**操作步骤：**

1. **创建 Feature**：Key = `cta_button_text`，类型 = `string`

2. **创建规则 1 — A 方案（50%）：**
   - 优先级: `10`
   - 条件: 灰度百分比 → `50`
   - 命中时: 开启，返回值 = `立即购买`

3. **创建规则 2 — B 方案（兜底）：**
   - 优先级: `100`
   - 条件: 空
   - 命中时: 开启，返回值 = `马上抢购`

> 💡 灰度百分比基于用户 ID 的哈希值，相同用户每次请求结果一致，不会随机跳变。

---

### 场景 7：JSON 配置下发

**背景：** `theme_config` 需要下发主题配色方案，不同环境用不同配色。

**操作步骤：**

1. **创建 Feature**：Key = `theme_config`，类型 = `json`

2. **dev 环境规则**：
   - 条件: 空
   - 命中时: 开启，返回值 = `{"primary":"#ff0000","bg":"#fff"}`

3. **prod 环境规则**：
   - 条件: 空
   - 命中时: 开启，返回值 = `{"primary":"#1890ff","bg":"#f5f5f5"}`

> 不同环境的规则互不干扰，这就是为什么要分环境管理。

---

### 场景 8：多条件组合 — VIP 用户 + iOS + 高版本

**背景：** `premium_feature` 仅对 VIP 用户中使用 iOS 且版本 ≥ 3.0 的人开放。

**操作步骤：**

1. 创建规则，添加 3 个条件：
   - 条件 1: 用户白名单 → VIP 用户 ID 列表
   - 条件 2: 平台 → `ios`
   - 条件 3: 版本约束 → `>=3.0.0`

三个条件之间是 AND 关系，必须同时满足才命中。

---

### 场景 9：不同环境不同策略

**背景：** `dark_mode` 功能在不同环境的策略不同。

| 环境 | 策略 |
|------|------|
| dev | 全部开启（方便开发调试） |
| staging | 白名单开启（QA 团队测试） |
| prod | 灰度 10%，逐步放量 |

**操作：** 在每个环境下分别创建不同的规则。规则是 **按环境隔离** 的，dev 的规则不会影响 prod。

---

### 场景 10：处理遗留功能的下线

**背景：** 旧功能 `legacy_export` 要逐步下线，先关闭新用户，再逐步关闭老用户。

**操作步骤：**

1. **阶段一 — 新用户关闭，老用户保持**：
   - 规则 1（优先级 0）: 用户白名单 → 老用户 ID 列表，命中时 = 开启
   - 规则 2（优先级 100）: 空条件，命中时 = 关闭

2. **阶段二 — 逐步缩减老用户名单**：
   - 编辑规则 1，逐步从白名单中移除已迁移的用户

3. **阶段三 — 全量关闭**：
   - 删除规则 1，仅留基线"关闭"规则

## 12. 用户级覆盖

用户覆盖允许个别用户自行设置某个 Feature 的状态，**优先级高于所有规则**。

这在以下场景有用：
- 用户在 App 设置页里手动开关了某个功能
- 客服需要为某个用户单独开启/关闭功能
- 开发人员需要在生产环境为自己开启调试功能

### 通过 API 设置覆盖

```bash
# 为用户 alice 强制开启 dark_mode
curl -X PUT http://localhost:8080/api/v1/override \
  -H "Content-Type: application/json" \
  -d '{
    "app_id": 1,
    "env_id": 2,
    "feature_id": 1,
    "user_id": "alice",
    "enabled": true,
    "value": ""
  }'
```

### 删除覆盖（回退到规则求值）

```bash
curl -X DELETE "http://localhost:8080/api/v1/override?app_id=1&env_id=2&feature_id=1&user_id=alice"
```

### 查看用户的所有覆盖

```bash
curl "http://localhost:8080/api/v1/overrides?app_id=1&env_id=2&user_id=alice"
```

## 13. 注意事项与 FAQ

### Q: 为什么我创建了 Feature 但客户端查询显示关闭？

**A:** Feature 创建后默认没有规则，没有规则 = 默认关闭。你需要至少创建一条规则。

### Q: 为什么我的规则没有生效？

**检查清单：**
1. 规则的 **"启用规则"** 开关是否打开？
2. 规则的 **环境** 是否与客户端请求的 `env_key` 匹配？
3. 是否有更高优先级（数字更小）的规则先命中了？
4. 该用户是否有 **覆盖 (Override)**，覆盖优先级高于所有规则。

### Q: 灰度百分比为什么某个用户一直命中/不命中？

**A:** 这是预期行为。灰度不是随机的，是基于 `hash(featureKey + userID) % 100` 的确定性计算。相同用户对同一个 Feature 的命中结果始终一致，确保用户体验稳定。

### Q: 多个条件之间是什么关系？

**A:** 在条件编辑器中添加的多个条件之间是 **AND（且）** 的关系。即所有条件都满足才算命中。

如果需要 OR（或）关系，可以拆成多条规则（同优先级），或通过 API 使用 `{"op":"or","items":[...]}` 格式。

### Q: 删除应用会怎样？

**A:** **级联删除**。删除应用时，其下的所有环境、Feature、规则、覆盖都会被一并删除。操作不可撤销，请谨慎。

### Q: 删除环境会怎样？

**A:** 该环境下的所有规则和覆盖会被级联删除。其他环境不受影响。

### Q: 什么是"基线规则"？

**A:** 基线规则是一条**空条件（match all）** 的规则，放在**最低优先级**（数字最大）。它的作用是：当没有任何定向规则命中时，作为兜底使用。

**强烈建议**每个 Feature 在每个环境下都设置一条基线规则。

### Q: Feature Key / App Key / Env Key 可以修改吗？

**A:** 不可以。这些标识一旦创建就是永久的，因为客户端代码里会硬编码这些 key。如果需要改，只能删除重建。

### Q: 如何安全地在生产环境操作？

**最佳实践：**
1. 先在 `dev` 环境配置和验证规则
2. 通过 **特性查询** 页面模拟不同用户验证
3. 确认无误后在 `staging` 环境配置相同规则
4. 最后在 `prod` 环境配置
5. 上线后通过 **操作审计** 确认操作正确

### Q: 优先级应该怎么设置？

**推荐规范：**

| 优先级范围 | 用途 |
|-----------|------|
| 0-9 | 紧急覆盖（如 Kill Switch） |
| 10-29 | 白名单规则 |
| 30-49 | 灰度规则 |
| 50-69 | 平台/版本定向 |
| 70-89 | 属性定向 |
| 90-100 | 基线规则（兜底） |

> 留出间隔方便后续插入新规则，避免修改已有规则的优先级。

## 附录：客户端接入

### 客户端查询 API

客户端（App/网页/服务器）通过一个 HTTP GET 请求获取所有 Feature 状态：

```
GET /api/v1/features?app_key=my_game&env_key=prod&user_id=alice&version=2.1.0&platform=ios&attr_region=cn
```

**参数说明：**

| 参数 | 必填 | 说明 |
|------|------|------|
| `app_key` | ✅ | 应用标识 |
| `env_key` | ✅ | 环境标识 |
| `user_id` | 否 | 用户 ID（灰度/白名单需要） |
| `version` | 否 | 客户端版本号 |
| `platform` | 否 | 平台（ios/android/windows/web 等） |
| `attr_*` | 否 | 自定义属性，前缀 `attr_`（如 `attr_region=cn`） |

**响应示例：**

```json
{
  "dark_mode": {
    "enabled": true,
    "value": ""
  },
  "welcome_text": {
    "enabled": true,
    "value": "欢迎使用全新版本！"
  },
  "new_checkout": {
    "enabled": false
  }
}
```

客户端根据返回的 `enabled` 和 `value` 决定功能是否显示以及显示什么内容。
